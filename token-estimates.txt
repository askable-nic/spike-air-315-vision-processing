  The likely explanation is that the moment merging/batching logic changed
  behavior. Experiment 2's lower thresholds produced more fine-grained moments,
  but many of them got merged together into longer interaction spans (e.g., the
  big 20000ms-28533ms interaction in exp 2 worth 13,500 tokens). This
  consolidation means fewer distinct before/after frame pairs needed per batch,
  but each pair covers more ground. The LLM then produces fewer discrete events
  because it's seeing merged spans rather than individual micro-interactions.

  The estimated token budget went up (188,950 vs 141,700) even though fewer
  images were sent — suggesting the token accounting got tighter in exp 2 (it
  now tracks token_budget / token_budget_used / token_budget_usage_pct which exp
   1 didn't have). The higher token estimate per moment is from the merged

⏺ Yes, it's overestimating. Here's why:

  When moments merge (line 1612), their token costs are summed:
  estimated_tokens=prev.estimated_tokens + m.estimated_tokens

  But when a merged moment is actually sent to the LLM (lines 996-1003), it
  still only produces 2 frames (one before, one after) regardless of how many
  constituent moments were merged:
  before_ts = max(0, moment.time_start_ms - 250)
  after_ts = moment.time_end_ms + 250

  So if 5 interaction moments merge into one, the estimate is 5 * 750 = 3,750 
  tokens, but the actual cost is still ~750 tokens (2 ROI crops).

  Looking at experiment 2's data, the big merged interaction at 20000ms-28533ms
  estimates 13,500 tokens — that's 18 original moments merged together. But it
  only sends 2 ROI crops (~750 actual tokens). That's an 18x overestimate for
  that single moment.

  The fix should update estimated_tokens after merging to reflect the actual
  frame cost of the merged result, not the sum of constituents. For a merged
  interaction with visual_change, it should be tokens_roi_pair (750), not the
  additive sum. Want me to fix that?
